## Overview

A fast-paced, real-time 2D top-down multiplayer battle royale game built with vanilla HTML5, Canvas 2D, and JavaScript ES Modules. It integrates Appwrite as a backend engine for real-time multiplayer synchronization, matchmaking (Solo/Duo), and sublobby chat. It features two game modes: **Standard Survival** (traditional battle royale with loot, weapons, zone) and **RPG-MOBA Sub-mode** (featuring character classes, level-up progression, spells, items shop, and custom base spawns).

## Components

The architecture is highly decoupled, centered around a shared game state module (`src/state.js`).

### 1. Centralized State Manager
- **Purpose:** Stores the shared global memory, active players, obstacles, bullets, UI configuration, input keys, joystick statuses, and RPG settings.
- **Location:** `src/state.js`
- **Pattern:** Singleton/State Store pattern.

### 2. Main Entry Point & Bootstrapping
- **Purpose:** Binds UI screen events, handles responsiveness, runs the continuous `requestAnimationFrame` game loop, and triggers initialization of peripheral sub-systems (audio, shop, customizer).
- **Location:** `src/main.js`

### 3. Controls & Inputs Handler
- **Purpose:** Registers keyboard/mouse inputs for desktop and renders/manages dual virtual joysticks for mobile screens.
- **Location:** `src/controls.js`

### 4. Player & Entities Model
- **Purpose:** Defines the `Player` class, player statistics, weapons mechanics (ammo counts, spread, damage, cooldowns, reload times), spell casting (Q/E abilities), experience (XP/level-up) dynamics, and deterministic map obstacle/loot generation.
- **Location:** `src/player.js`

### 5. Core Game Rules & Bot AI
- **Purpose:** Orchestrates high-frequency updates, bot pathfinding/behavior, collision resolution, damage calculation, bullet traversal, and game status changes.
- **Location:** `src/game.js`

### 6. Canvas 2D Rendering Engine
- **Purpose:** Renders player sprites, weapon overlays, bullet animations, spell effects (frost rings, meteors, shields), blood decals, minimap overlays, deterministic red zones, and fog of war.
- **Location:** `src/renderer.js`

### 7. Appwrite Multiplayer Network Connector
- **Purpose:** Performs database CRUD operations and real-time WebSocket subscriptions to sync lobby rooms, players' coordinates, health states, and sublobby chat logs.
- **Location:** `src/network.js`

### 8. UI Layer
- **Purpose:** Handles vanilla DOM interactions, hud updates (health, gold, level, class choice), cosmetic customizers, shop purchases, and screens switching.
- **Location:** `src/ui.js`

### 9. Audio System
- **Purpose:** Plays context-driven audio cues (firing, punching, healing, level-up) using the browser's Web Audio / DOM audio capabilities.
- **Location:** `src/audio.js`

### 10. Global Constants
- **Purpose:** Configures map sizing, weapon metadata, RPG classes, and key UI attributes.
- **Location:** `src/constants.js`

## Data Flow

1. **Initialization:**
   - `window.onload` in `main.js` binds UI elements, loads customizers, triggers Appwrite websocket handshakes via `network.js`, and sets up input listeners in `controls.js`.
   
2. **Game Loop (`requestAnimationFrame`):**
   - **Update Phase:** `gameLoop()` calling `updateGame()` in `game.js` processes player motion, handles obstacles collisions, moves bullets, updates AI bot directions/actions, and ticks down active spell cooldowns/effects.
   - **Render Phase:** `drawGame()` in `renderer.js` redraws the floor grid, players, foliage/rocks, active bullets/spell particles, minimap, and HUD interface.
   
3. **Multiplayer Sync Protocol (120ms tick):**
   - High-speed player states (coordinates, rotation, current weapon, health) are published to the Appwrite Database every 120ms via `network.js` and broadcast to peers through Appwrite's Realtime channel to ensure interpolation.

## Conventions

- **Language & Comments:** Code is written in standard ES6 JavaScript. High-level structure and variables use English, while documentation comments, HUD string resources, and debug labels are predominantly written in **Czech**.
- **State Management:** All components access and modify a single global, reactive `state` export from `src/state.js` instead of using complex message-passing networks.
- **Canvas Rendering:** Decoupled drawing routines that execute sequentially based on layering priority (Floor -> Dead Players -> Bullets -> Active Players -> Foliage Tops -> HUD).

## Technical Debt

- [ ] **Hardcoded Require statement:** `player.js` line 526 imports `game.js` using `require('./game.js')`, which is not natively supported in standard ES Modules and causes runtime Vite exceptions or build failures.
- [ ] **Global State Mutability:** The global `state` object can be modified directly by any component, which might lead to race conditions during rapid state transitions.
- [ ] **DB Real-Time Sync Frequencies:** Database synchronization runs at a heavy 120ms interval per player, creating high database overhead that could scale poorly under multi-room pressure.
